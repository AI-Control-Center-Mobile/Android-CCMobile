# Спецификация приложения

## 1. Назначение приложения

`AIControlCentreAndroid` — локальный Android-клиент для работы с несколькими LLM-провайдерами без собственного backend-сервера приложения. Приложение хранит ключи провайдеров на устройстве, организует работу по проектам и тредам, отправляет запросы в модели, показывает потоковую генерацию ответов, позволяет сравнить ответы двух моделей в рамках одного треда и отображает диагностическую информацию по доступным квотам провайдеров.

Ключевой принцип приложения: пользователь работает внутри локального workspace на устройстве, а сетевое взаимодействие идёт напрямую с API провайдеров.

## 2. Общая функциональная карта

- Инициализация приложения и подключение провайдеров по API-ключам.
- Локальное хранение ключей, проектов, тредов, сообщений и кэша каталога моделей.
- Управление проектами как верхним уровнем организации работы.
- Управление тредами внутри проекта.
- Быстрый запуск нового треда из самого свежего проекта.
- Чат с выбранной моделью внутри треда.
- Потоковое отображение ответа ассистента.
- Выбор модели для чата и compare-сценариев.
- Сравнение двух моделей по одному prompt внутри текущего треда.
- Сохранение победившего ответа из compare обратно в историю треда.
- Просмотр сохранённых ключей и диагностик провайдеров.
- Очистка ключей или полное удаление локальных данных приложения.
- Автоматическое создание демонстрационного workspace при первом сохранённом ключе и пустой локальной базе.

## 3. Экраны и разделы

### 3.1. Setup / Initialize

- Назначение: первый экран приложения при полном отсутствии сохранённых ключей провайдеров.
- Что пользователь видит:
  - форму выбора провайдера;
  - поле ввода API-ключа;
  - карточку с уже настроенными провайдерами;
  - блок с пояснением, что ключи хранятся локально и в зашифрованном виде;
  - кнопку сохранения ключа;
  - кнопку перехода в `Projects` после появления хотя бы одного сохранённого ключа.
- Какие действия доступны:
  - выбрать одного из провайдеров: `OpenRouter`, `Groq`, `SiliconFlow`;
  - ввести API-ключ;
  - сохранить ключ;
  - перейти в основной раздел приложения после успешной конфигурации.
- Переходы и зависимости:
  - является стартовым экраном, если в хранилище нет ни одного ключа;
  - после сохранения ключа переводит пользователя в `Projects`.
- Важные состояния/ограничения:
  - пустой ключ не сохраняется;
  - сохранение идёт только локально;
  - экран открывается повторно, если из `Settings` удалить последний ключ.

### 3.2. Projects

- Назначение: список рабочих пространств, организующих переписки по темам или задачам.
- Что пользователь видит:
  - форму создания проекта с полем названия;
  - список уже созданных проектов;
  - нижнюю навигацию `Projects / New Chat / Settings`.
- Какие действия доступны:
  - создать новый проект;
  - открыть карточку проекта.
- Переходы и зависимости:
  - открытие карточки проекта ведёт в `Project Detail`;
  - нижняя навигация ведёт в `New Chat` и `Settings`.
- Важные состояния/ограничения:
  - если проектов нет, показывается пустое состояние;
  - проекты отображаются по последней активности.

### 3.3. Project Detail

- Назначение: обзор одного проекта и связанных с ним тредов.
- Что пользователь видит:
  - кнопку возврата;
  - заголовок проекта;
  - агрегированную сводку по проекту: число сообщений, суммарные оценочные расходы, время последней активности;
  - кнопку создания нового треда;
  - список недавних тредов с превью последнего сообщения, счётчиком сообщений и накопленной стоимостью;
  - нижнюю навигацию верхнего уровня.
- Какие действия доступны:
  - вернуться назад;
  - создать новый тред;
  - открыть существующий тред.
- Переходы и зависимости:
  - новый тред и выбор существующего треда ведут в `Thread`;
  - нижняя навигация ведёт в `Projects`, `New Chat`, `Settings`.
- Важные состояния/ограничения:
  - если тредов нет, показывается пустое состояние;
  - новый тред создаётся с дефолтным названием `New Thread`.

### 3.4. New Chat

- Назначение: быстрый старт нового треда без ручного входа в конкретный проект.
- Что пользователь видит:
  - либо пустое состояние с предложением сначала создать проект;
  - либо карточку целевого проекта, в который будет создан новый тред.
- Какие действия доступны:
  - перейти в `Projects`, если проектов ещё нет;
  - создать новый тред в автоматически выбранном проекте.
- Переходы и зависимости:
  - после создания треда сразу открывается экран `Thread`;
  - если проектов нет, переход ведёт в `Projects`.
- Важные состояния/ограничения:
  - используется только один проект-назначение: первый в списке проектов, то есть самый свежий по `updatedAt`;
  - если проект успели удалить до создания треда, пользователь получает ошибку и должен вернуться к списку проектов.

### 3.5. Thread

- Назначение: основная рабочая зона переписки с моделью внутри выбранного треда.
- Что пользователь видит:
  - кнопку возврата;
  - кнопки `Compare` и `Models`;
  - номер треда;
  - chip'ы с текущим провайдером, выбранной моделью и статусом синхронизации каталога моделей;
  - историю сообщений;
  - composer для ввода prompt;
  - потоковый блок ответа ассистента во время генерации.
- Какие действия доступны:
  - ввести сообщение и отправить его;
  - открыть compare-сценарий;
  - открыть выбор модели;
  - копировать содержимое каждого сообщения;
  - просматривать markdown-ответы ассистента.
- Переходы и зависимости:
  - `Compare` открывает экран сравнения моделей для этого же треда;
  - `Models` открывает `Model Picker` в режиме chat;
  - возврат ведёт назад к проекту или предыдущему экрану.
- Важные состояния/ограничения:
  - без выбранной модели отправка невозможна;
  - пустой prompt не отправляется;
  - если каталога моделей нет и синхронизация ещё идёт, показывается loading;
  - если модели не загрузились, экран показывает ошибку и кнопку повторной синхронизации;
  - при смене модели между ответами ассистента в истории показывается уведомление о переключении модели;
  - ответы ассистента отображают провайдера, модель, latency и оценочную стоимость, если эти данные пришли от провайдера;
  - для сообщений ассистента поддерживается markdown-рендеринг, включая заголовки, таблицы, кодовые блоки и выделенные reasoning-блоки `<think>`.

### 3.6. Compare

- Назначение: параллельное сравнение двух моделей на одном prompt внутри текущего треда.
- Что пользователь видит:
  - кнопку возврата;
  - маркер режима `Compare A/B`;
  - поле prompt;
  - кнопки выбора `Model A` и `Model B`;
  - кнопку запуска сравнения;
  - две независимые карточки результатов `A` и `B`;
  - кнопку продолжения с победителем.
- Какие действия доступны:
  - изменить prompt для сравнения;
  - выбрать две модели;
  - запустить compare;
  - повторить только неудавшийся ответ конкретной модели;
  - сохранить победивший ответ в тред и вернуться в обычный чат.
- Переходы и зависимости:
  - открывается из `Thread`;
  - для выбора каждой модели использует `Model Picker` в отдельных режимах;
  - после выбора победителя возвращает пользователя в `Thread` и передаёт туда победившую модель как активную.
- Важные состояния/ограничения:
  - для compare нужно минимум две доступные compare-модели;
  - сравнение не запускается с пустым prompt;
  - сравнение не запускается, если выбраны одинаковые модели;
  - prompt по умолчанию подставляется из последнего пользовательского сообщения в треде;
  - модель `A` по умолчанию наследуется из текущей chat-модели, если она доступна;
  - каждая сторона compare может завершиться успешно или с ошибкой независимо;
  - кнопка `Continue with winner` становится активной только после полного завершения выбранного ответа;
  - если compare запускался на prompt, которого ещё нет последним пользовательским сообщением в треде, перед сохранением победителя этот prompt сначала добавляется в историю.

### 3.7. Model Picker

- Назначение: выбор модели для обычного чата или для compare-слотов.
- Что пользователь видит:
  - кнопку возврата;
  - маркер режима: `CHAT MODEL`, `COMPARE A`, `COMPARE B`;
  - строку поиска;
  - список моделей с названием, техническим id, провайдером и отметкой выбранной модели.
- Какие действия доступны:
  - искать модели по label, provider id и model id;
  - выбрать модель;
  - повторить загрузку каталога при ошибке.
- Переходы и зависимости:
  - открывается из `Thread` и `Compare`;
  - после выбора модели возвращает значение на предыдущий экран.
- Важные состояния/ограничения:
  - список фильтруется по режиму: chat-режим показывает chat-модели, compare-режимы — compare-модели;
  - если сеть недоступна, экран использует локальный кэш моделей, если он есть;
  - без ключей провайдера экран показать рабочий список моделей не сможет.

### 3.8. Settings

- Назначение: управление ключами провайдеров, просмотр квот и выполнение локально-разрушающих операций.
- Что пользователь видит:
  - форму выбора провайдера и сохранения нового ключа;
  - список уже сохранённых ключей в замаскированном виде;
  - блок `Provider quota and diagnostics`;
  - кнопку обновления диагностик;
  - кнопку очистки всех сохранённых ключей;
  - `Danger Zone` с полным удалением локальных данных.
- Какие действия доступны:
  - выбрать провайдера и сохранить новый ключ;
  - удалить отдельный ключ;
  - очистить все ключи;
  - обновить данные по квотам;
  - полностью очистить все локальные данные.
- Переходы и зависимости:
  - находится в нижней навигации верхнего уровня;
  - если после удаления ключей не остаётся ни одного ключа, приложение переводит пользователя обратно в `Setup`.
- Важные состояния/ограничения:
  - `Clear saved keys` удаляет только ключи, но не трогает проекты, треды и сообщения;
  - `Wipe all local data` удаляет ключи, проекты, треды, сообщения и сохранённые preference-данные;
  - все разрушительные действия подтверждаются диалогом;
  - список квот зависит от провайдеров, для которых реально сохранены ключи.

## 4. Пользовательские сценарии

### 4.1. Первый запуск и разблокировка приложения

1. Пользователь открывает приложение.
2. Если ключей провайдеров ещё нет, стартует экран `Setup`.
3. Пользователь выбирает провайдера и вводит API-ключ.
4. Приложение сохраняет ключ локально в защищённом хранилище.
5. После появления хотя бы одного ключа пользователь переходит в `Projects`.
6. Если локальная база была пустой, приложение автоматически создаёт демонстрационные проекты, треды и сообщения.

### 4.2. Создание нового проекта и нового треда

1. Пользователь открывает `Projects`.
2. Вводит название нового проекта.
3. Создаёт проект.
4. Открывает карточку проекта.
5. Нажимает `Start New Thread`.
6. Приложение создаёт новый тред и открывает экран `Thread`.

### 4.3. Быстрый старт нового треда через New Chat

1. Пользователь открывает вкладку `New Chat`.
2. Если проектов нет, приложение предлагает перейти в `Projects`.
3. Если проекты есть, приложение выбирает самый свежий проект.
4. Пользователь нажимает `Start new thread`.
5. Новый тред создаётся в выбранном проекте и сразу открывается.

### 4.4. Обычный чат в треде

1. Пользователь открывает тред.
2. Приложение загружает или обновляет каталог моделей.
3. Пользователь выбирает модель напрямую из текущего состояния или через `Models`.
4. Пользователь вводит prompt и отправляет его.
5. Приложение сохраняет пользовательское сообщение в локальную базу.
6. Ответ ассистента отображается потоково по мере генерации.
7. После завершения ответ сохраняется в локальную историю вместе с метаданными провайдера, модели, latency и стоимости.

### 4.5. Смена модели внутри треда

1. Пользователь нажимает `Models`.
2. Открывается `Model Picker` в chat-режиме.
3. Пользователь ищет и выбирает модель.
4. Экран возвращается в `Thread`.
5. Выбранная модель становится активной для следующих сообщений.

### 4.6. Сравнение двух моделей

1. Пользователь находится в треде и нажимает `Compare`.
2. Экран compare подставляет последний пользовательский prompt, если он есть.
3. Пользователь при необходимости меняет prompt и выбирает `Model A` и `Model B`.
4. Приложение запускает две независимые потоковые генерации.
5. Пользователь наблюдает оба ответа и их метаданные.
6. Если один из ответов упал, пользователь может повторить только эту сторону compare.
7. Пользователь выбирает победителя через `Continue with winner`.
8. Победивший ответ сохраняется в тред, compare закрывается, а победившая модель становится активной моделью треда.

### 4.7. Работа с ключами и диагностикой провайдеров

1. Пользователь открывает `Settings`.
2. Добавляет ключи для одного или нескольких провайдеров.
3. Просматривает список сохранённых ключей в замаскированном виде.
4. Запрашивает обновление квот и диагностик.
5. При необходимости удаляет один ключ, все ключи или полностью очищает локальные данные.

## 5. Функции и возможности

### 5.1. Работа с провайдерами

- Поддерживаются три провайдера: `OpenRouter`, `Groq`, `SiliconFlow`.
- Для каждого провайдера приложение хранит отдельный ключ.
- Ключи можно добавлять по одному, комбинировать между собой и удалять независимо.
- Стартовый доступ к основным функциям приложения открывается при наличии хотя бы одного ключа любого провайдера.

### 5.2. Локальный workspace

- Проекты и треды хранятся локально в базе Room.
- Проект служит контейнером для группы тредов.
- Тред хранит полную историю сообщений.
- Последняя активность треда обновляет как сам тред, так и родительский проект.
- История сообщений сохраняется между сессиями приложения.
- При пустой базе и первом доступном ключе автоматически добавляется демо-наполнение:
  - 2 проекта;
  - 3 треда;
  - стартовые пользовательские и ассистентские сообщения.

### 5.3. Каталог моделей

- Приложение запрашивает каталоги моделей у всех провайдеров, для которых есть ключи.
- Локально сохраняется объединённый кэш моделей.
- При успешной загрузке кэш полностью обновляется.
- Если сеть недоступна, приложение пытается использовать сохранённый кэш.
- Свежий кэш моделей считается пригодным до 6 часов.
- Модели сортируются по провайдеру и названию.

### 5.4. Чат

- Отправка сообщений выполняется прямо из треда.
- В запрос уходит история предыдущей переписки этого треда.
- Ответы ассистента приходят потоково и видны пользователю до полного завершения.
- После завершения ответ сохраняется как полноценное сообщение в истории.
- Визуально различаются пользовательские и ассистентские сообщения.
- Для каждого ассистентского сообщения могут отображаться:
  - провайдер;
  - модель;
  - latency;
  - estimated cost.
- Пользователь может скопировать содержимое любого сообщения.

### 5.5. Compare

- Compare запускается только внутри конкретного треда.
- Для одного prompt параллельно запрашиваются две модели.
- Обе стороны compare отображаются независимо и могут иметь разный статус:
  - ожидание;
  - потоковая генерация;
  - завершено;
  - ошибка.
- Пользователь может повторить запрос только для упавшей стороны.
- Побеждающий ответ переносится в историю треда без ручного копирования.

### 5.6. Представление ответов

- Ответы ассистента отображаются как markdown-контент.
- Поддерживаются:
  - заголовки;
  - абзацы;
  - code blocks;
  - таблицы;
  - горизонтальные разделители;
  - reasoning-блоки внутри `<think>...</think>`.
- Для превью тредов reasoning-блоки не показываются, а markdown уплощается до краткого текста.

### 5.7. Диагностика и квоты

- Для `OpenRouter` приложение может запрашивать live-диагностику текущего ключа.
- Для `OpenRouter` показываются значения типа:
  - остаток лимита;
  - дневное использование;
  - время reset;
  - label ключа, если его вернул API.
- Для `SiliconFlow` показываются live-данные о балансе.
- Для `Groq` показывается не live API-диагностика, а последний сохранённый snapshot rate-limit заголовков.
- Snapshot `Groq` появляется только после хотя бы одного успешного сетевого запроса к Groq через приложение.

### 5.8. Управление данными

- Можно удалить один конкретный ключ провайдера.
- Можно очистить все ключи, не трогая историю работы.
- Можно полностью удалить все локальные данные приложения.
- Полная очистка убирает:
  - ключи провайдеров;
  - проекты;
  - треды;
  - сообщения;
  - preference-данные.

## 6. Настройки, разрешения и системное поведение

- Приложение запрашивает только системное разрешение `INTERNET`.
- Код не содержит запроса на push-уведомления, доступ к контактам, камере, микрофону, геолокации или файловой системе пользователя.
- Приложение не поднимает фоновый worker-процесс, периодическую синхронизацию или очередь фоновых задач; сеть используется по запросу пользователя при открытии экранов и выполнении действий.
- API-ключи хранятся в `EncryptedSharedPreferences` с `MasterKey`.
- Локальные preference-данные хранятся через `DataStore`.
- Проекты, треды, сообщения и каталог моделей хранятся в Room.
- `allowBackup=false`: приложение не рассчитано на стандартное Android backup-восстановление через системный backup.
- Стартовый маршрут определяется состоянием ключей:
  - ключей нет — `Setup`;
  - есть хотя бы один ключ — `Projects`.
- При удалении последнего ключа приложение сбрасывает пользователя обратно в setup-flow.

## 7. Интеграции и внешние зависимости

- `OpenRouter`
  - прямой доступ к каталогу моделей;
  - прямые chat/completions-запросы через OpenAI-compatible endpoint;
  - получение диагностик текущего ключа.
- `Groq`
  - прямой доступ к каталогу моделей;
  - прямые chat/completions-запросы через OpenAI-compatible endpoint;
  - сохранение rate-limit snapshot из HTTP-заголовков ответов.
- `SiliconFlow`
  - прямой доступ к каталогу моделей;
  - прямые chat/completions-запросы через OpenAI-compatible endpoint;
  - получение данных о пользовательском балансе.
- Сетевое взаимодействие выполняется напрямую с устройства через `OkHttp` и `Retrofit`.
- Собственный сервер приложения в текущем коде не используется.

## 8. Ограничения и особые условия

- Без сохранённого ключа хотя бы одного провайдера пользователь не может перейти к рабочим разделам приложения.
- Если список моделей не удалось загрузить и локального кэша нет, chat-, compare- и picker-сценарии блокируются до успешной повторной синхронизации.
- Для compare требуется минимум две доступные compare-модели.
- Compare не допускает использование одной и той же модели в обоих слотах.
- Диагностика `Groq` зависит от ранее накопленного snapshot и не гарантирует live-состояние.
- Для `OpenRouter` код явно предупреждает о более жёстких rate limits у моделей, чьи id оканчиваются на `:free`.
- При удалении всех ключей проекты и сообщения могут оставаться локально, если пользователь выбрал очистку только ключей, а не полную очистку данных.
- В пользовательском интерфейсе текущей версии не обнаружены экраны удаления проектов или тредов, хотя соответствующие repository-методы существуют.

## 9. Неоднозначные или неполностью подтвержденные наблюдения

- Приложение позиционируется как AI control workspace, но код не навязывает конкретную бизнес-доменную специализацию проектов и тредов; фактически это универсальный локальный клиент для работы с LLM-переписками.
- Текущий mapper сетевого каталога помечает все загруженные модели как одновременно пригодные и для chat, и для compare. По коду нельзя подтвердить более тонкие ограничения конкретных моделей со стороны провайдера.
- В `DataStore` объявлен ключ `LAST_PROJECT_ID`, но в текущем пользовательском потоке его использование не подтверждается.
